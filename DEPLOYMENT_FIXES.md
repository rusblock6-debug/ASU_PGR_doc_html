### БАГ #1: Line endings в entrypoint.sh
**Проблема:** Docker контейнер не запускался с ошибкой `exec /entrypoint.sh: no such file or directory`

**Причина:** Файл `entrypoint.sh` имел Windows line endings (CRLF) вместо Unix (LF)

**Решение:**
```powershell
# Остановили контейнеры
docker compose -f docker-compose.rag.yml -p pgr-rag down

# Конвертировали CRLF → LF через временный файл
Get-Content ai-bot/entrypoint.sh -Raw | ForEach-Object { $_ -replace "`r`n", "`n" } | Set-Content -NoNewline ai-bot/entrypoint_fixed.sh
Remove-Item ai-bot/entrypoint.sh -Force
Move-Item ai-bot/entrypoint_fixed.sh ai-bot/entrypoint.sh

# Пересобрали контейнер
docker compose -f docker-compose.rag.yml -p pgr-rag build ai-bot
docker compose -f docker-compose.rag.yml -p pgr-rag up -d
```

**Статус:** ✅ Исправлено. Контейнер запустился, модель phi4-mini загружена, FastAPI работает на порту 8000.

### БАГ #2: Индексация зависает на долгое время
**Проблема:** POST запрос на `/api/index?mode=full` не возвращает ответ очень долго (10+ минут)

**Причина:** 
- Индексация обрабатывает файлы **последовательно** (один за другим)
- Для каждого `.md`, `.yaml`, `.json`, `.html` файла вызывается Phi-4-mini через HTTP
- Каждый вызов Phi-4 занимает 5-10 секунд
- В репозитории **сотни** таких файлов → общее время 10-50 минут
- HTTP запрос висит без промежуточных ответов

**Код проблемы в `ai-bot/main.py`:**
```python
for file_info in files:  # ← ПОСЛЕДОВАТЕЛЬНО!
    chunks = await CodeChunker.chunk_file_async(file_info['path'], content, use_phi4=True)
    # ↑ Каждый .md/.yaml файл → 5-10 сек через Phi-4
```

**Решение (варианты):**

**Вариант 1: Отключить Phi-4 для первой индексации (быстро)**
- Изменить `use_phi4=False` для первого запуска
- Потом можно переиндексировать с Phi-4

**Вариант 2: Добавить параллельную обработку (правильно)**
- Использовать `asyncio.gather()` для параллельной обработки файлов
- Ограничить количество одновременных запросов к Phi-4 (например, 5)

**Вариант 3: Добавить прогресс-бар и логирование**
- Возвращать промежуточные результаты через WebSocket
- Показывать прогресс индексации в реальном времени

**Статус:** ⏳ В процессе. Жду команды какой вариант применить.

**ОБНОВЛЕНИЕ:** ✅ Применён вариант 2 - параллельная обработка

**Изменения в `ai-bot/main.py`:**
- Добавлена функция `process_file()` для асинхронной обработки
- Файлы обрабатываются батчами по 5 штук параллельно
- Используется `asyncio.gather()` для параллельного выполнения
- Логирование прогресса после каждого батча

**Ожидаемая скорость:**
- Было: 1 файл → 5 сек (последовательно)
- Стало: 5 файлов → 7-8 сек (параллельно)
- **Ускорение: ~4x раза**
- Полная индексация: **5-10 минут** вместо 20-30

**Нагрузка на систему:**
- CPU: 30-50% (16 ядер справятся)
- RAM: +500 MB (Phi-4 уже в памяти)
- Можно продолжать работать

**Контейнер перезапущен с новым кодом.**

### БАГ #3: Три одновременных процесса индексации
**Проблема:** Пользователь запустил команду индексации **3 раза** в разных терминалах одновременно

**Последствия:**
- Phi-4 перегружен (обрабатывает 3x файлов параллельно)
- ChromaDB получает конфликтующие записи
- Индексация в 3 раза медленнее
- Возможна порча данных в базе

**Решение:**
```powershell
# Перезапустили контейнер, убив все процессы индексации
docker restart pgr-ai-bot
```

**Статус:** ✅ Исправлено. Контейнер перезапущен, все процессы остановлены.

**Очистка базы данных:**
```bash
# Удалили мусорные данные из ChromaDB
docker exec pgr-ai-bot python -c "import chromadb; client = chromadb.PersistentClient(path='/app/chroma_data'); client.delete_collection('pgr_docs')"
```
Результат: База очищена (было 145 документов → стало 0)
