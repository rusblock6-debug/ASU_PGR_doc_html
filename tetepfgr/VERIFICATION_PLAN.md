# План проверки полноты и точности документации АСУ ПГР

## Цель
Создать ПОЛНОЕ и ТОЧНОЕ руководство пользователя, описывающее ВСЕ функции системы, которые видит оператор. Каждый шаг должен соответствовать реальной системе (коду), а не бумажной документации.

## Критерии качества
1. **ПОЛНОТА**: Все экраны, кнопки, функции описаны
2. **ТОЧНОСТЬ**: Каждый шаг соответствует реальному UI в коде
3. **ЯСНОСТЬ**: Инструкции для НЕ-программистов, максимально простые
4. **СТИЛЬ**: Повелительное наклонение (ОТКРОЙТЕ, НАЖМИТЕ, ВЫБЕРИТЕ)

## Источники истины (в порядке приоритета)
1. **ИСХОДНЫЙ КОД** frontend компонентов - ГЛАВНЫЙ источник
2. Реальные endpoint'ы API из кода backend
3. Документация в documentation/ - только как дополнение

## Алгоритм проверки (для каждого раздела)

### Шаг 1: Анализ кода компонента
```
1. Найти все Vue/React компоненты (.vue, .tsx, .jsx файлы)
2. Извлечь маршруты (routes)
3. Найти все UI элементы:
   - Кнопки (button, @click, onClick)
   - Формы (input, select, textarea)
   - Таблицы (table, v-for, map)
   - Модальные окна (dialog, modal)
   - Меню навигации
4. Записать реальные названия кнопок/полей
5. Записать последовательность действий
```

### Шаг 2: Сверка с data.json
```
1. Открыть соответствующий раздел в data.json
2. Проверить КАЖДЫЙ шаг инструкции:
   - Существует ли такая кнопка в коде?
   - Правильное ли название поля?
   - Верная ли последовательность действий?
   - Все ли опции перечислены?
3. Исправить несоответствия СРАЗУ в data.json
```

### Шаг 3: Поиск недостающих функций
```
1. Сравнить список всех найденных в коде функций с описанием в data.json
2. Если функция есть в коде, но нет в data.json - ДОБАВИТЬ новый раздел
3. Приоритет добавления:
   - Основные пользовательские сценарии
   - Часто используемые функции
   - Критичные для работы операции
```

### Шаг 4: Проверка стиля инструкций
```
1. Убедиться что все инструкции в повелительном наклонении
2. Проверить форматирование предупреждений (⚠️)
3. Проверить списки (— для маркированных)
4. Убрать технические термины или объяснить их
```

## Методология поиска в коде

### Для Vue компонентов:
```bash
# Найти все компоненты
grep -r "\.vue$" components/*/src/

# Найти маршруты
grep -r "path:" components/*/src/ --include="*.js" --include="*.ts"

# Найти кнопки
grep -r "@click\|onClick" components/*/src/ --include="*.vue" --include="*.tsx"

# Найти формы
grep -r "v-model\|ref(" components/*/src/ --include="*.vue"

# Найти модальные окна
grep -r "dialog\|modal" components/*/src/ --include="*.vue" -i
```

### Для React компонентов:
```bash
# Найти компоненты
grep -r "\.tsx$\|\.jsx$" components/*/src/

# Найти маршруты
grep -r "Route\|navigate" components/*/src/ --include="*.tsx" --include="*.ts"

# Найти обработчики
grep -r "onClick\|onSubmit" components/*/src/ --include="*.tsx"
```

## Правила исправления несоответствий

### Если название кнопки отличается:
```
Было: "Нажмите кнопку Создать"
Стало: "Нажмите кнопку «+ Добавить»"  // точное название из кода
```

### Если последовательность действий неверна:
```
Было: "Выберите статус, затем нажмите Сохранить"
Стало: "Нажмите «Сохранить», затем выберите статус"  // правильный порядок
```

### Если отсутствует важный шаг:
```
Добавить новый объект в steps array с правильным порядком
```

### Если технический термин:
```
Было: "Выберите endpoint API"
Стало: "Выберите адрес сервера"  // понятный пользователю термин
```
