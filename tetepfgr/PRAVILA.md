# Правила написания документации АСУ ПГР

## 1. РАБОТА С КОДОМ И ПРОЕКТОМ

### 1.1. Структура проекта
- ✅ Проект находится в: `C:\Сторонние\АСУ_ПГР_(репо)\`
- ✅ Основные папки проекта:
  - `components/` - компоненты системы (микросервисы)
  - `infrastructure/` - инфраструктура
  - `tetepfgr/` - документация пользователя

### 1.2. Работа с документацией
- ✅ Документация находится ТОЛЬКО в папке `tetepfgr/`
- ✅ Редактируем только файлы:
  - `tetepfgr/data.json` - основная документация (инструкции, описания)
  - `tetepfgr/directory_data.json` - справочники параметров
  - `tetepfgr/admin.html` - интерфейс редактирования (НЕ ТРОГАЕМ без запроса)
- ✅ НЕ создаем новые файлы без явного запроса
- ✅ НЕ создаем markdown-файлы для "резюме" или "отчетов"

### 1.3. Изучение кода для документации
- ✅ Для понимания функционала изучаем код в папках:
  - `components/` - смотрим как работают компоненты системы
  - `infrastructure/` - смотрим инфраструктуру
- ✅ НЕ редактируем код в `components/` и `infrastructure/`
- ✅ Используем код только для ПОНИМАНИЯ функционала
- ✅ Описываем в документации то, что РЕАЛЬНО есть в коде

### 1.4. Путь работы
```
1. Смотрим скриншот интерфейса
2. Изучаем код компонента (если нужно понять логику)
3. Описываем функционал простым языком
4. Сохраняем в data.json или directory_data.json
```

## 5. СТРУКТУРА ОПИСАНИЯ

### 5.1. Формат каждого шага
```
Название действия
├─ Краткое описание (что делаем и зачем)
├─ Пошаговая инструкция:
│  1. Первое действие
│  2. Второе действие
│     — Подробность (с тире)
│     — Еще подробность
│  3. Третье действие
├─ ⚠️ ВАЖНО: критичные моменты
└─ Скриншоты
```

### 5.2. Правила нумерации
- ✅ Каждый шаг начинается с цифры: "1.", "2.", "3."
- ✅ Подробности начинаются с тире: "—"
- ✅ Вложенные пункты с отступом
- ✅ Пустая строка между шагами для читаемости

### 5.3. Глаголы действия
- ✅ Каждый шаг начинается с глагола в повелительном наклонении:
  - "Откройте раздел..."
  - "Нажмите кнопку..."
  - "Выберите из списка..."
  - "Введите значение..."
  - "Дождитесь загрузки..."
  - "Просмотрите результат..."

## 6. РАБОТА СО СКРИНШОТАМИ

### 6.1. Количество и качество
- ✅ Один шаг = 1-3 скриншота максимум
- ✅ Скриншоты должны быть последовательными
- ✅ Каждый скриншот показывает конкретное действие
- ✅ Скриншот должен быть четким и читаемым

### 6.2. Именование файлов
- ✅ Используем формат: `[номер_шага].[номер_скриншота]_[описание].png`
- ✅ Примеры:
  - `1.1_ecosystem_main.png` - шаг 1, скриншот 1
  - `1.2_asu_pgr_card.png` - шаг 1, скриншот 2
  - `2.1_create_model.png` - шаг 2, скриншот 1
  - `2.2_model_form.png` - шаг 2, скриншот 2

### 6.3. Нумерация скриншотов
- ✅ ВСЕГДА указывай номера скриншотов при описании
- ✅ Формат: "Скриншот 1.1: описание" (шаг.номер)
- ✅ Номера соответствуют порядку в массиве images в JSON
- ✅ Пример:
  ```
  Скриншот 1.1 (1.1_ecosystem_main.png): главный экран экосистемы
  Скриншот 1.2 (1.2_asu_pgr_card.png): карточка АСУ ПГР выделена
  Скриншот 1.3 (1.3_login_form.png): окно авторизации
  ```

### 6.4. Структура папок
```
screenshots/
├── quickstart/          # Быстрый старт
├── instructions/        # Пошаговые инструкции
├── main-windows/        # Главные окна системы
├── vehicles/            # Справочник: Оборудование
├── statuses/            # Справочник: Статусы
├── cargo/               # Справочник: Виды груза
├── horizons/            # Справочник: Горизонты
├── places/              # Справочник: Места работ
├── tags/                # Справочник: Метки позиционирования
├── sections/            # Справочник: Участки
├── work-orders/         # Наряд-задания
└── trips/               # Рейсы
```

## 7. ФОРМАТ ИНСТРУКЦИЙ В JSON

### 7.1. Структура шага
```json
{
  "text": "Название действия",
  "details": [
    "1. Первое действие",
    "2. Второе действие",
    "   — Подробность",
    "   — Еще подробность",
    "3. Третье действие",
    "",
    "⚠️ ВАЖНО:",
    "— Критичный момент 1",
    "— Критичный момент 2"
  ],
  "images": [
    "screenshots/раздел/имя_файла_1.png",
    "screenshots/раздел/имя_файла_2.png"
  ],
  "horizontal": false
}
```

### 7.2. Параметр horizontal
- `"horizontal": false` - скриншоты в столбик (по умолчанию)
- `"horizontal": true` - скриншоты в ряд (для сравнения "до/после")

## 9. ОПИСАНИЕ ЭЛЕМЕНТОВ ИНТЕРФЕЙСА

### 9.1. Для каждого поля формы указываем:

```
Название поля *
├─ Обязательное: да/нет (звездочка * = обязательное)
├─ Тип данных: текст/число/дата/выбор из списка
├─ Формат: например "YYYY-MM-DD" для даты
├─ Ограничения: например "от 1 до 100"
├─ Пример: конкретное значение
└─ Что произойдет: результат заполнения
```

**Пример:**
```
Гаражный номер *
├─ Обязательное: да
├─ Тип: текст
├─ Формат: до 50 символов
├─ Пример: "СА-001", "4", "22"
└─ Используется в названии техники
```

### 9.2. Для каждой кнопки указываем:

```
Название кнопки
├─ Расположение: где находится
├─ Цвет/стиль: для идентификации
├─ Когда активна: условия
├─ Что произойдет: результат нажатия
└─ Сообщение: что увидит пользователь
```

**Пример:**
```
Кнопка "Сохранить"
├─ Расположение: внизу формы справа
├─ Цвет: оранжевый
├─ Активна: когда все обязательные поля заполнены
├─ Результат: техника добавляется в таблицу
└─ Сообщение: "Объект добавлен в таблицу"
```

## 10. ПРЕДУПРЕЖДЕНИЯ И ВАЖНЫЕ МОМЕНТЫ

### 10.1. Когда добавлять предупреждения

Используем символ ⚠️ для:
- Обязательных действий
- Ограничений системы
- Возможных ошибок
- Критичных параметров
- Необратимых действий

### 10.2. Формат предупреждений

```
⚠️ ВАЖНО:
— Первый важный момент
— Второй важный момент
— Третий важный момент
```

### 10.3. Примеры предупреждений

```
⚠️ ВАЖНО:
— Без заполнения плотности груза система не сможет рассчитать вес
— Системные статусы нельзя удалить
— Изменение модели повлияет на все единицы техники этой модели
```
