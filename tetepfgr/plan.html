⚙️ 3. Принцип работы — по шагам (без домыслов)

📥 Этап 1. Индексация — как бот «читает» репозиторий

1 Сканирование ВСЕГО репозитория (кроме мусора: .git, node_modules, __pycache__, картинки, логи)
2 Разбиение на чанки: код → функции/классы, JSON → объекты, документация → параграфы
3 Embedding → превращаем каждый чанк в вектор (числа)
4 Сохраняем в ChromaDB — это и есть «база знаний»

Что важно: индексация НЕ происходит сама собой. Вы или CI/CD запускаете команду:
curl -X POST /api/index?mode=full
После этого бот готов отвечать.

▼

💬 Этап 2. Как бот отвечает на вопрос пользователя

1 Пользователь пишет вопрос в чат → попадает на ai-bot
2 Проверка Redis: если такой вопрос уже был — мгновенный ответ
3 Embedding вопроса → вектор
4 Поиск в ChromaDB → топ-20 похожих чанков
5 Reranker (bge-reranker-v2-m3) → выбирает топ-4 самых релевантных
6 Отправляем эти 4 чанка в Ollama (qwen2.5:3b) с инструкцией: «отвечай ТОЛЬКО на основе этих данных»
7 Нейросеть генерирует ответ + ссылки на источники
⏱️ Время ответа: 12–18 секунд (первый раз). Повторные вопросы — мгновенно (кэш).
🔗 Пример ответа: «Согласно функции get_vehicles() в файле enterprise-service/vehicles.py, строка 42…»

📸 4. Цифровой снимок (Snapshot) — вы управляете процессом

Зачем? Вы должны знать, что именно попадёт в базу знаний, прежде чем запустить индексацию.

1️⃣ Создать снимок
curl -X POST /api/snapshot/create
→ получаете JSON-файл со списком всех файлов, хешей, предупреждениями.
2️⃣ Проверить снимок
curl /api/snapshot/latest
Вы видите, какие папки и файлы будут индексироваться, а какие исключены.
3️⃣ Индексация по снимку
curl -X POST "/api/index?snapshot=latest"
Только после этой команды бот строит ChromaDB.
✅ Что даёт снимок:
— Вы НЕ теряете контроль: сначала посмотрели, потом запустили.
— Снимки можно сохранять в Git (воспроизводимость).
— Можно сравнить два снимка (что изменилось в проекте).

📂 5. Что попадает в индекс, а что нет (чётко)

✅ Сканируется ВСЁ, КРОМЕ:
‣ .git — история коммитов (не нужно)
‣ node_modules — чужие библиотеки
‣ __pycache__ — кэш Python
‣ logs/, chroma_data/, ollama_data/, redis_data/
‣ картинки (.jpg, .png, .gif)
‣ бинарники (.exe, .so, .dll, .pyc)
‣ служебные файлы (.env, .gitignore, .dockerignore)
✅ ВСЕ остальные файлы — в индексе:
‣ Весь код (.py, .go, .ts, .tsx, .js)
‣ Инфраструктура (Dockerfile, docker-compose.yml)
‣ SQL (миграции, initdb)
‣ JSON, YAML, Markdown, HTML-документы
‣ ТЗ, тест-кейсы, документация
📌 Почему так:
Код всегда актуален — это единственный источник правды.
Исключается только мусор, который не несёт смысловой нагрузки.
🧠 Итог по объёму: ~1500–2000 файлов → ~20 000–30 000 чанков → ChromaDB легко тянет.

🔄 5.1 Транзакционная индексация (чтобы не сломать базу)

🔴 ОБЯЗАТЕЛЬНОЕ ТРЕБОВАНИЕ: при обновлении индекса использовать временную коллекцию. 
def reindex_incremental():
    # 1. Создаём временную коллекцию
    temp_collection = client.create_collection("temp", temporary=True)
    
    # 2. Индексируем изменённые файлы во временную коллекцию
    for file in changed_files:
        chunks = chunk_file(file)
        temp_collection.add(chunks)
    
    # 3. Атомарная замена
    main_collection.delete(where={"file_path": {"$in": changed_files}})
    main_collection.add(temp_collection.get())
    
    # 4. Удаляем временную
    client.delete_collection("temp")

🔄 7. Обновление знаний — инкрементально, без перестройки всего мира

После изменений в коде или документации достаточно запустить:

curl -X POST "/api/reindex?mode=incremental" -H "X-API-Key: ваш-ключ"
1 Бот вычисляет MD5-хеши всех файлов
2 Сравнивает с хешами в ChromaDB
3 Для изменившихся файлов → удаляет старые векторы, создаёт новые
4 Новые файлы → добавляются, удалённые → векторы уходят
⚡ Время инкрементального обновления: 1–3 минуты (вместо 15–35 минут полной индексации).
🔁 Переименовали папку? — reindex подхватит.
🆕 Добавили новый микросервис? — достаточно добавить его в components/ (или куда вы его положили) и запустить reindex.

🤖 10. CI/CD — всё само, вы только пишете код

Пример .gitlab-ci.yml (отдел настраивает один раз):

stages:
  - snapshot
  - index

create_snapshot:
  stage: snapshot
  script: curl -X POST ${BOT_URL}/api/snapshot/create
  only: [main, develop]

index_full:
  stage: index
  script: curl -X POST "${BOT_URL}/api/index?mode=full"
  only: [tags]                # при создании тега (релиз)

index_incremental:
  stage: index
  script: curl -X POST "${BOT_URL}/api/reindex?mode=incremental"
  only: [schedules]           # каждую ночь в 3:00
  variables:
    SCHEDULE: "0 3 * * *"
📦 Push в main → создаётся снимок (индексация НЕ трогается)
🏷️ Создание тега → полная переиндексация (релиз)
⏰ Ночной шедулер → инкрементальное обновление
